# P6: VC Revocation & Status Management
(การเพิกถอนและการจัดการสถานะเอกสาร)

> Two of the three sub-flows from `docs/User_Journey/transcript/P6.md` (Holder-initiated and Issuer-initiated revocation) are reused here verbatim. **The Single-Use / double-spend-prevention case is absent from this diagram** — consistent with an ID Card credential not being a single-use document type.

## Case 1 — Holder requests cancellation/suspension
(กรณี Holder ขอ ยกเลิก/อายัด เอกสาร)

1. **[Holder]** Case: Holder requests to cancel/suspend a document (กรณี Holder ขอ ยกเลิก/อายัด เอกสาร)
2. **[Digital Wallet]** Receive the command and request identity verification (รับคำสั่ง และร้องขอการยืนยันตัวตน)
3. **[Holder]** Enter PIN/Biometric to verify identity (ใส่ PIN / Biometric เพื่อยืนยันตัวตน)
4. **[Digital Wallet]** Send the cancellation/suspension request, with evidence, to the Issuer (ส่งคำขอยกเลิก/อายัดเอกสารพร้อมหลักฐาน ไปให้ Issuer)
5. **[Issuer]** Check the validity of the key evidence (PoP) (ตรวจสอบความถูกต้องของหลักฐานกุญแจ (PoP))
6. **[Issuer]** Approve and instruct the VC's status to be updated (อนุมัติ และสั่งอัปเดตสถานะของเอกสารรับรอง (VC))
7. **[VC Status Registry]** Record the document status as Cancelled/Suspended (บันทึกสถานะเอกสารเป็น ยกเลิก/อายัด)
8. **[Audit Trail]** Record the status-change history (บันทึกประวัติการปรับสถานะเอกสาร)
9. **[Issuer]** Notify the result of the VC status update (แจ้งผลการอัปเดตสถานะของเอกสารรับรอง (VC))
10. **[Digital Wallet]** Command the secure chip to safely destroy that document's key (สั่งการชิปนิรภัย ทำลายกุญแจของเอกสารนั้นๆ ทิ้งอย่างปลอดภัย)
11. **[Holder]** Acknowledge the result of the cancellation/suspension (รับทราบผลของการยกเลิก/อายัดเอกสาร)
12. **[Holder]** End process — Holder-initiated case (สิ้นสุดกระบวนการ กรณี Holder เป็นผู้สั่งการ)

## Case 2 — Issuer orders revocation/suspension
(กรณี Issuer สั่ง พิกถอน/ระงับ การใช้เอกสาร)

1. **[Issuer]** Case: Issuer orders revocation/suspension of document use (กรณี Issuer สั่ง เพิกถอน/ระงับการใช้เอกสาร)
2. **[Issuer]** Instruct the VC's status to be updated (สั่งอัปเดตสถานะของเอกสารรับรอง (VC))
3. **[VC Status Registry]** Record the document status as Cancelled/Suspended (บันทึกสถานะ เอกสารเป็น ยกเลิก/อายัด)
4. **[Audit Trail]** Record the status-change history (บันทึกประวัติการปรับสถานะ เอกสาร)
5. **[Issuer]** Send a notification of the revocation/suspension (ส่งการแจ้งเตือนการเพิกถอน/ระงับ ใช้เอกสาร)
6. **[Holder]** Acknowledge that the document has been revoked/suspended (รับทราบว่า เอกสารถูกเพิกถอน/ระงับ)
7. **[Holder]** End process — Issuer-initiated case (สิ้นสุดกระบวนการกรณี Issuer เป็นผู้สั่งการ)

## Summary

P6 (ID Card) covers the two ways an ID-Card credential's lifecycle can end before normal expiry/renewal. (1) **Holder-initiated**: the Holder authenticates locally (PIN/Biometric), the Wallet forwards a cancel/suspend request with PoP evidence to the Issuer, the Issuer approves and orders the status change, the VC Status Registry persists "Cancelled/Suspended" while the Audit Trail logs it, and — only after the Issuer confirms — the Wallet has the secure chip permanently destroy that credential's private key. (2) **Issuer-initiated**: the Issuer can unilaterally revoke/suspend the credential, recording the status change and notifying the Holder, who simply acknowledges. Unlike the generic transcript flow (`docs/User_Journey/transcript/P6.md`), there is no Single-Use case here, since an ID Card is a reusable, long-lived credential rather than a one-time-use document. Both cases converge on the same pattern: status persisted centrally (VC Status Registry), changes audited (Audit Trail), and on-device key material destroyed via the secure chip — consistent with the non-extractable key boundary in `src/services/crypto/**`.
