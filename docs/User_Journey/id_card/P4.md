## Flow steps

1. **[Holder]** Start process (เริ่มกระบวนการ)
2. **[Holder]** Contact the Verifier by scanning a QR Code / tapping NFC / clicking a link (ติดต่อกับ Verifier โดยการสแกน QR Code / แตะ NFC / คลิ๊กลิงก์)
3. **[Verifier]** Send the document request, specifying the required data fields (ส่งคำขอเอกสาร พร้อมระบุฟิลด์ข้อมูลที่ต้องการ)
4. **[Digital Wallet]** Request the Verifier's public key (ร้องขอกุญแจสาธารณะของ Verifier)
5. **[DID Resolver]** Look up and return the Verifier's public key via its DID Document (ค้นหาและคืนค่ากุญแจสาธารณะของ Verifier ผ่าน DID Document)
6. **[Digital Wallet]** Request a check on the Verifier's trustworthiness (ร้องขอตรวจความน่าเชื่อถือของ Verifier)
7. **[Trust Registry]** Confirm the trust status of the Verifier (ยืนยันสถานะความน่าเชื่อถือของผู้ตรวจเอกสาร (Verifier))
8. **[Digital Wallet]** Select the data per the request — Selective Disclosure (เลือกข้อมูลตามที่ถูกร้องขอ (Selective Disclosure))
9. **[Digital Wallet]** Display the consent screen (แสดงหน้าจอขอความยินยอม (Consent))
10. **[Holder]** Review the data to be presented and tap Accept (ตรวจสอบข้อมูลที่จะนำไปสำแดงและกดยอมรับ)
11. **[Holder]** Enter PIN/Biometric to confirm identity (ระบุ PIN / Biometric เพื่อยืนยันตัวตน)
12. **[Digital Wallet]** Command the secure chip to generate proof of key possession (สั่งการชิปนิรภัย เพื่อสร้างหลักฐานการครอบครองกุญแจ)
13. **[Digital Wallet]** Build the VP document from the requested data, with Proof of Possession (สร้างเอกสาร VP ตามข้อมูลที่ร้องขอ พร้อม Proof of Possession)
14. **[Verifier]** Receive the VP document to continue verification (รับเอกสาร VP เพื่อตรวจสอบความถูกต้องต่อไป)
15. **[Verifier]** End process (สิ้นสุดกระบวนการ)

## Summary

P4 is the online Verifiable Presentation (VP) flow for the ID-Card scenario: the Holder initiates contact with a Verifier (QR/NFC/link), the Verifier replies with a request for specific data fields, and the Wallet resolves the Verifier's public key (via the DID Resolver) and trust status (via the Trust Registry) before proceeding. The Wallet then applies selective disclosure to the requested fields, surfaces a consent screen, and — only after the Holder explicitly accepts and authenticates with PIN/Biometric — invokes the secure chip to produce a Proof of Possession and assembles the signed VP, which is sent to the Verifier to close the loop. This mirrors the hardware-signing and biometric sign-time-gate boundary enforced in `src/services/crypto/**`.
