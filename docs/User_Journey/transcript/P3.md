# P3: VC Holding & Lifecycle Management
(การถือครอง จัดการเอกสาร และบริหารกุญแจ)

Source: `P3 Transcript - Swimlane Diagram v.1.xlsx`

## Lanes (actors / systems)

| Lane | Role |
|---|---|
| VCGA | Governance authority (lane present, no steps in this diagram) |
| Holder | The user who owns the wallet; receives/acknowledges notifications |
| Issuer | Verifies the old credential and issues the renewed one |
| Digital Wallet | Drives key rotation, requests renewal, stores/cleans up credentials |
| Wallet Provider | (lane present, no steps in this diagram) |
| Trust Registry | Confirms the Issuer's trust status |
| Schema Registry | Provides the credential template for the new VC |
| VC Status Registry | Checks/updates credential status (suspend old, activate new) |
| Audit Trail | Records renewal history |

## Flow steps

1. **[Digital Wallet]** Start process (เริ่มกระบวนการ)
2. **[Digital Wallet]** Start key-rotation process when the key expires — see Figure 1 (เริ่มกระบวนการเปลี่ยนกุญแจเมื่อกุญแจหมดอายุ, รูปที่ 1)
3. **[Digital Wallet]** Generate a new key (`did:key`) to prepare for the new VC (สร้างกุญแจดอกใหม่ (did:key) เตรียมสำหรับเอกสาร VC ใบใหม่)
4. **[Digital Wallet]** Send the request, attaching the old VC, to the relevant Issuer — see Figure 2 (ส่งคำขอ พร้อมแนบ VC ใบเก่า ไปยัง Issuer ที่เกี่ยวข้อง, รูปที่ 2)
5. **[Issuer]** Receive the request and start the verification process on the old VC (รับคำขอ และนำ VC ใบเก่าไปเริ่มกระบวนการตรวจสอบ)
6. **[Trust Registry]** Confirm the trust status of the Issuer — see Figure 3 (ยืนยันสถานะความน่าเชื่อถือของผู้ออกเอกสาร (Issuer), รูปที่ 3)
7. **[VC Status Registry]** Check that the old VC is still in normal status, not suspended (ตรวจสอบว่า VC ใบเก่ายังคงสถานะปกติ ไม่ถูกระงับ)
8. **[Issuer]** Verification succeeded — confirm the old VC is valid and trustworthy — see Figure 4 (ตรวจสอบสำเร็จและยืนยันว่า VC ใบเก่าถูกต้องและเชื่อถือได้, รูปที่ 4)
9. **[Issuer]** Send a new-credential issuance offer (ส่งข้อเสนอการออกเอกสารใหม่)
10. **[Digital Wallet]** Receive the offer and send a new VC request together with Proof-of-Possession (PoP) (รับข้อเสนอและส่งคำขอ VC ใหม่ พร้อมพิสูจน์การครอบครองกุญแจ (PoP))
11. **[Issuer]** Receive the request and verify the validity of the PoP (รับคำขอและตรวจสอบความถูกต้องของการครอบครองกุญแจ (PoP))
12. **[Schema Registry]** Retrieve the document template (ดึงแม่แบบเอกสาร)
13. **[Issuer]** Create the new VC with the new key and apply a digital signature (สร้าง VC ใบใหม่ด้วยกุญแจใหม่ พร้อมลงลายมือชื่อดิจิทัล)
14. **[VC Status Registry]** Revoke the old VC and activate the new VC ("Active") — see Figure 5 (เพิกถอน VC ใบเก่า เปิดใช้งาน VC ใบใหม่ ("Active"), รูปที่ 5)
15. **[Audit Trail]** Record the renewal history (บันทึกประวัติการต่ออายุเอกสาร)
16. **[Issuer]** Deliver the new VC document (ส่งมอบเอกสาร VC ใบใหม่)
17. **[Digital Wallet]** Verify correctness and store the new VC in the wallet — see Figure 6 (ตรวจสอบความถูกต้อง และจัดเก็บเอกสาร VC ใบใหม่ลงกระเป๋า, รูปที่ 6)
18. **[Digital Wallet]** Delete the old VC document and destroy the old key — see Figure 7 (ลบเอกสาร VC ใบเก่า และทำลายกุญแจดอกเก่าทิ้ง, รูปที่ 7)
19. **[Digital Wallet]** Send a notification message to the user (ส่งข้อความแจ้งเตือนผู้ใช้งาน)
20. **[Holder]** Acknowledge the notification (รับทราบการแจ้งเตือน)
21. **[Holder]** End process (สิ้นสุดกระบวนการ)

## Summary

P3 covers proactive credential lifecycle management — specifically key/credential renewal triggered when a VC's key expires. The Wallet generates a fresh `did:key`, then submits the old VC plus a renewal request to the Issuer. The Issuer cross-checks the old VC's validity via the Trust Registry (issuer trust) and VC Status Registry (suspension state) before issuing a renewal offer. The Wallet proves possession of its new key (PoP), the Issuer pulls the schema template, signs a new VC, and atomically revokes the old one / activates the new one in the VC Status Registry while logging the event to the Audit Trail. Finally the Wallet validates and stores the new VC, deletes the old VC and destroys its old private key, and notifies the Holder — closing the loop on secure key rotation without requiring the Holder to re-prove identity from scratch (cf. the non-extractable key boundary rules in `src/services/crypto/**`).
