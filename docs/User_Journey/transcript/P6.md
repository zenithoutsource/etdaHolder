# P6: VC Revocation & Status Management
(การเพิกถอนและการจัดการสถานะเอกสาร)

Source: `P6 Transcript - Swimlane Diagram v.1 - Copy 15.47.48.xlsx`

This diagram has three independent sub-flows (cases) sharing the same lanes.

## Lanes (actors / systems)

| Lane | Role |
|---|---|
| VCGA | Governance authority (lane present, no steps in this diagram) |
| Issuer | Verifies the request, approves, instructs status update, notifies |
| Holder | Initiates cancel/suspend, authenticates, acknowledges outcomes |
| Digital Wallet | Relays commands, prompts authentication, triggers secure-chip key destruction |
| Verifier | (Single-Use case only) reports that a credential has already been consumed |
| VC Status Registry | Persists the document status (e.g., Cancelled/Suspended, Used) |
| Audit Trail | Records the status-change history |

---

## Case 1 — Holder requests cancellation/suspension
(กรณี Holder ขอ ยกเลิก/อายัด เอกสาร)

1. **[Holder]** Case: Holder requests to cancel/suspend a document (กรณี Holder ขอ ยกเลิก/อายัด เอกสาร)
2. **[Digital Wallet]** Receive the command and request identity verification (รับคำสั่ง และร้องขอการยืนยันตัวตน)
3. **[Holder]** Enter PIN/Biometric to verify identity (ใส่ PIN / Biometric เพื่อยืนยันตัวตน)
4. **[Digital Wallet]** Send the cancellation/suspension request, with evidence, to the Issuer (ส่งคำขอยกเลิก/อายัดเอกสารพร้อมหลักฐาน ไปให้ Issuer)
5. **[Issuer]** Check the validity of the key evidence (PoP) (ตรวจสอบความถูกต้องของหลักฐานกุญแจ (PoP))
6. **[Issuer]** Approve and instruct the VC's status to be updated (อนุมัติ และสั่งอัปเดตสถานะของเอกสารรับรอง (VC))
7. **[VC Status Registry]** Record the document status as Cancelled/Suspended (บันทึกสถานะเอกสารเป็น ยกเลิก/อายัด)
8. **[Audit Trail]** Record the status-change history (บันทึกประวัติการปรับสถานะเอกสาร)
9. **[Issuer]** Notify the result of the VC status update (แจ้งผลการอัปเดตสถานะของเอกสารรับรอง (VC))
10. **[Digital Wallet]** Command the secure chip to safely destroy that document's key (สั่งการชิปนิรภัย ทำลายกุญแจของเอกสารนั้นๆ ทิ้งอย่างปลอดภัย)
11. **[Holder]** Acknowledge the result of the cancellation/suspension (รับทราบผลของการยกเลิก/อายัดเอกสาร)
12. **[Holder]** End process — Holder-initiated case (สิ้นสุดกระบวนการ กรณี Holder เป็นผู้สั่งการ)

## Case 2 — Issuer orders revocation/suspension
(กรณี Issuer สั่ง พิกถอน/ระงับ การใช้เอกสาร)

1. **[Issuer]** Case: Issuer orders revocation/suspension of document use (กรณี Issuer สั่ง เพิกถอน/ระงับการใช้เอกสาร)
2. **[Issuer]** Instruct the VC's status to be updated (สั่งอัปเดตสถานะของเอกสารรับรอง (VC))
3. **[VC Status Registry]** Record the document status as Cancelled/Suspended (บันทึกสถานะ เอกสารเป็น ยกเลิก/อายัด)
4. **[Audit Trail]** Record the status-change history (บันทึกประวัติการปรับสถานะ เอกสาร)
5. **[Issuer]** Send a notification of the revocation/suspension (ส่งการแจ้งเตือนการเพิกถอน/ระงับ ใช้เอกสาร)
6. **[Holder]** Acknowledge that the document has been revoked/suspended (รับทราบว่า เอกสารถูกเพิกถอน/ระงับ)
7. **[Holder]** End process — Issuer-initiated case (สิ้นสุดกระบวนการกรณี Issuer เป็นผู้สั่งการ)

## Case 3 — Single-Use document (prevent double-spending)
(กรณีเอกสาร Single-Use (ป้องกัน double spending))

1. **[Verifier]** Case: Single-Use document — prevent double-spending (กรณีเอกสาร Single-Use (ป้องกัน double spending))
2. **[Verifier]** Notify that the document has already been used (แจ้งว่าใช้งานเอกสารแล้ว)
3. **[Digital Wallet]** Command the secure chip to destroy the key and mark the document as Used — Self-Cleanup (สั่งการชิปนิรภัย ทำลายกุญแจ และปรับสถานะเอกสารเป็นใช้แล้ว (Self-Cleanup))
4. **[Holder]** Acknowledge that the document's right has been used (รับทราบว่าเอกสารถูกใช้สิทธิ์ไปแล้ว)
5. **[Holder]** End process — Single-Use case (สิ้นสุดกระบวนการ กรณี Single-Use)

## Summary

P6 covers the three ways a credential's lifecycle can end before normal expiry/renewal. (1) **Holder-initiated**: the Holder authenticates locally (PIN/Biometric), the Wallet forwards a cancel/suspend request with PoP evidence to the Issuer, the Issuer approves and orders the status change, the VC Status Registry persists "Cancelled/Suspended" while the Audit Trail logs it, and — only after the Issuer confirms — the Wallet has the secure chip permanently destroy that credential's private key. (2) **Issuer-initiated**: the Issuer can unilaterally revoke/suspend a credential, recording the status change and notifying the Holder, who simply acknowledges. (3) **Single-Use**: a Verifier that detects a credential has already been consumed reports it, prompting the Wallet to self-cleanup — destroying the key and marking the credential Used to prevent replay/double-spending. All three converge on the same pattern: status persisted centrally (VC Status Registry), changes audited (Audit Trail), and on-device key material destroyed via the secure chip — consistent with the non-extractable key boundary in `src/services/crypto/**`.
