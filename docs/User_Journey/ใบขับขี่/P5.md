# P5: VP Verification
(การตรวจสอบความน่าเชื่อถือของเอกสาร)

Source: `P5 DRIVING LICENSE - Flow-1_Swimlane Diagram v.1.3-1 - Copy.xlsx`

> Same VP-verification flow as `docs/User_Journey/transcript/P5.md` — content is identical (the Driving License scenario reuses the shared verification diagram).

## Lanes (actors / systems)

| Lane | Role |
|---|---|
| Holder | Sends the VP (via the Wallet) and receives the final verification outcome |
| Digital Wallet | Sends the VP with Proof of Possession to the Verifier |
| Verifier | Drives the entire verification: signature, trust, schema, status, PoP checks |
| Trust Registry | Confirms the trust status of the Issuer |
| Schema Registry | Provides the document's structural template for validation |
| VC Status Registry | Reports whether the document is suspended/revoked |
| DID Resolver | Resolves the Issuer's public key from its DID Document |
| Audit Trail | Records the successful verification |

## Flow steps

1. **[Digital Wallet]** Start process (เริ่มกระบวนการ)
2. **[Digital Wallet]** Send the presentation document (VP) together with Proof of Possession (PoP) (ส่งเอกสารสำแดง (VP) พร้อมหลักฐานการครอบครองกุญแจ (PoP))
3. **[Verifier]** Receive the VP document and start the verification process (รับเอกสารสำแดง (VP) และเริ่มกระบวนการตรวจสอบ)
4. **[Verifier]** Request the Issuer's public key (ร้องขอกุญแจสาธารณะของผู้ออกเอกสาร (Issuer))
5. **[DID Resolver]** Look up and return the Issuer's public key via its DID Document (ค้นหาและคืนค่ากุญแจสาธารณะของ Issuer ผ่าน DID Document)
6. **[Verifier]** Confirm the correctness of the Issuer's signature (ยืนยันความถูกต้องลายมือชื่อของผู้ออกเอกสาร (Issuer))
7. **[Trust Registry]** Check the trust status of the Issuer (ตรวจสอบสถานะความน่าเชื่อถือของผู้ออกเอกสาร (Issuer))
8. **[Verifier]** Confirm the trustworthiness of the Issuer (ยืนยันความน่าเชื่อถือของผู้ออกเอกสาร (Issuer))
9. **[Schema Registry]** Retrieve the document's structural template details (ดึงรายละเอียดแม่แบบโครงสร้างของเอกสาร)
10. **[Verifier]** Confirm the correctness of the document's structural template (ยืนยันความถูกต้องของแม่แบบโครงสร้างของเอกสาร)
11. **[VC Status Registry]** Check the document's status (ตรวจสอบสถานะของเอกสาร)
12. **[Verifier]** Confirm the document's status is neither suspended nor revoked (ยืนยันว่าสถานะของเอกสารไม่ถูกระงับ และไม่ถูกเพิกถอน)
13. **[Verifier]** Check the PoP to prove the document genuinely came from the Holder's device (ตรวจสอบ PoP เพื่อพิสูจน์ว่าเอกสารมาจากเครื่องของ Holder จริง)
14. **[Verifier]** Verification fully succeeded (ตรวจสอบสำเร็จครบถ้วน)
15. **[Audit Trail]** Record the successful verification history (บันทึกประวัติการตรวจสอบสำเร็จ)
16. **[Verifier]** Notify the result to the Digital Wallet (แจ้งผลกับไปยัง Digital Wallet)
17. **[Verifier]** Acknowledge and display the verification result (รับแจ้ง และแสดงผลการตรวจสอบ)
18. **[Holder]** Acknowledge the verification result (รับทราบผลการตรวจสอบ)
19. **[Holder]** End process (สิ้นสุดกระบวนการ)

## Summary

P5 is the Verifier-side counterpart to P4: once it receives a VP with Proof of Possession, the Verifier independently re-derives trust in the credential through four parallel checks — (a) resolving the Issuer's public key via the DID Resolver and validating the signature, (b) confirming the Issuer's trust status via the Trust Registry, (c) validating the document's structural template against the Schema Registry, and (d) confirming the document is neither suspended nor revoked via the VC Status Registry — then validates the PoP to bind the presentation to the Holder's device. On full success it logs the event to the Audit Trail, reports the result back to the Digital Wallet, and displays it; the Holder is finally informed of the outcome, completing the end-to-end trust chain that P4 (presentation) initiates.
