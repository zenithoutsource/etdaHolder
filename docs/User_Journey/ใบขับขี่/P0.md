## Flow steps

1. **[Issuer / Verifier]** Start process (เริ่มกระบวนการ)
2. **[Issuer / Verifier]** Prepare the system, generate `did:web` and a CSR (Certificate Signing Request) (เตรียมระบบ สร้าง did:web และ CSR (Certificate Signing Request))
3. **[Issuer / Verifier]** Submit the registration request to VCGA (ยื่นคำขอลงทะเบียนต่อ VCGA)
4. **[VCGA]** Verify qualifications and approve the registration (ตรวจสอบคุณสมบัติ และอนุมัติการลงทะเบียน) — รูปที่ 1
5. **[Trust Registry]** Record the Issuer/Verifier's trust status (บันทึกสถานะความน่าเชื่อถือของ Issuer/Verifier)
6. **[Schema Registry]** Record the Issuer/Verifier's document schema template (บันทึกแม่แบบโครงสร้างเอกสารของ Issuer/Verifier)
7. **[Audit Trail]** Record the registration-approval history (บันทึกประวัติการอนุมัติการลงทะเบียน)
8. **[VCGA]** Issue an X.509 Access Certificate (ออกใบรับรองการเข้าถึง (Access Certificate) แบบ X.509)
9. **[Issuer / Verifier]** Receive the result and the certificate (รับผลลัพธ์ และใบรับรอง)
10. **[Issuer / Verifier]** End process (สิ้นสุดกระบวนการ)

## Summary

P0 (Driving License) is the ecosystem-bootstrap flow shared identically across all three scenarios in this repo (`transcript`, `id_card`, `ใบขับขี่`): an organization wishing to act as Issuer/Verifier prepares its system, generates a `did:web` identity and a CSR, and submits a registration request to VCGA (the governance authority). VCGA verifies the applicant's qualifications and approves it, which fans out into three parallel registry writes — Trust Registry (trust status), Schema Registry (document schema template), and Audit Trail (approval history) — before VCGA issues an X.509 Access Certificate back to the Issuer/Verifier, completing its onboarding into the trusted ecosystem. This is the precondition for every later Driving-License-specific flow (issuance, presentation, verification, revocation): without a registered, certificate-bearing Issuer/Verifier recorded in the Trust and Schema Registries, no VC can be trusted by any party.
