# P2: Standard VC Issuance
(การขอออกเอกสาร VC อื่นๆ)

Source: `P2-DRIVING LICENSE Swimlane Diagram v.1.xlsx`

> Same standard-VC-issuance flow as `docs/User_Journey/transcript/P2.md` — content is identical (the Driving License scenario reuses the shared issuance diagram).

## Lanes (actors / systems)

| Lane | Role |
|---|---|
| VCGA | Governance authority (lane present, bordering Issuer's box visually; no distinct step of its own) |
| Issuer | Issues the requested Verifiable Credential (VC) |
| Holder | The user who owns the Digital Wallet and consents to actions |
| Digital Wallet | App that requests, receives, and stores the VC on the Holder's behalf |
| Wallet Provider | (lane present, no steps in this diagram) |
| Trust Registry | Confirms trustworthiness/registration status of Issuers |
| Schema Registry | Provides credential templates |
| VC Status Registry | Tracks and exposes credential status (e.g., Active) |
| Audit Trail | Records issuance history |

## Flow steps

1. **[Holder]** Start process (เริ่มกระบวนการ)
2. **[Issuer]** Request for credential (VC) — see Figures 1, 2, 3 (ร้องขอเอกสารรับรอง (VC), รูปที่ 1/2/3)
   - Sub-variants referenced: แบบ ก (Form A), แบบ ข (Form B), แบบ ค (Form C)
3. **[Digital Wallet]** Receive instruction and connect to the relevant Issuer system (รับคำสั่งและเชื่อมต่อระบบไปยัง Issuer ที่เกี่ยวข้อง)
4. **[Issuer]** Acknowledge connection and request the user to verify their identity (ตอบรับและร้องขอให้ผู้ใช้ยืนยันตัวตน)
5. **[Digital Wallet]** Display PID VC for identity verification — see Figure 4 (แสดง PID VC เพื่อยืนยันตัวตน, รูปที่ 4)
6. **[Holder]** Review the request and enter PIN/Biometric to give consent — see Figure 5 (ตรวจสอบคำขอ และระบุ PIN/Biometric เพื่อให้ความยินยอม, รูปที่ 5)
7. **[Digital Wallet]** Send the PID VC to the relevant Issuer — see Figure 6 (ดำเนินการส่ง PID VC ให้ Issuer ที่เกี่ยวข้อง, รูปที่ 6)
8. **[Issuer]** Receive the PID VC document and start the verification process (รับเอกสาร PID VC และเริ่มกระบวนการตรวจสอบ)
9. **[Trust Registry]** Confirm the trustworthiness of the PID Issuer — see Figure 7 (ยืนยันความน่าเชื่อถือของ PID Issuer, รูปที่ 7)
10. **[VC Status Registry]** Check the status of the PID VC document (ตรวจสอบสถานะของเอกสาร PID VC)
11. **[Issuer]** Confirm successful identity verification and send a credential-preparation offer (ยืนยันตัวตนสำเร็จ และส่งข้อเสนอเตรียมการออกเอกสาร)
12. **[Digital Wallet]** Generate a new key (`did:key`) for this credential (สร้างกุญแจดอกใหม่ (did:key) สำหรับเอกสารฉบับนี้)
13. **[Digital Wallet]** Send the credential request together with Proof-of-Possession (PoP) (ส่งคำขอเอกสาร พร้อมพิสูจน์ความเป็นเจ้าของกุญแจ (PoP))
14. **[Issuer]** Receive the request and verify the PoP (รับคำขอ และตรวจสอบ PoP)
15. **[Schema Registry]** Retrieve the document template (ดึงแม่แบบเอกสาร)
16. **[Issuer]** Create the VC document and apply a digital signature (สร้างเอกสาร VC และลงลายมือชื่อดิจิทัล)
17. **[VC Status Registry]** Record the document status as "Active" (พร้อมใช้งาน) (บันทึกสถานะเอกสารว่า "พร้อมใช้งาน (Active)")
18. **[Audit Trail]** Record the issuance history — see Figure 8 (บันทึกประวัติการออกเอกสาร, รูปที่ 8)
19. **[Issuer]** Deliver the credential (ส่งมอบเอกสารรับรอง)
20. **[Digital Wallet]** Receive the credential and start its own verification mechanism (รับเอกสารรับรอง และเริ่มกลไกการตรวจสอบ)
21. **[Trust Registry]** Confirm that the Issuer is properly registered and trustworthy — see Figure 9 (ยืนยันว่า Issuer ได้ขึ้นทะเบียนถูกต้องและเชื่อถือได้, รูปที่ 9)
22. **[Digital Wallet]** Verification successful — accept the document into the wallet (ตรวจสอบสำเร็จ จึงยอมรับเอกสารลงกระเป๋า)
23. **[Digital Wallet]** End process (สิ้นสุดกระบวนการ)

## Summary

P2 is the standard VC-issuance flow: the Holder, via the Digital Wallet, first proves identity to the Issuer by presenting an existing PID VC (which the Issuer cross-checks against the Trust Registry and VC Status Registry). After identity is confirmed, the Wallet generates a fresh `did:key` and submits a credential request with Proof-of-Possession; the Issuer fetches the appropriate template from the Schema Registry, builds and signs the new VC, marks it Active in the VC Status Registry, logs the issuance in the Audit Trail, and delivers it. The Wallet then independently re-verifies the Issuer's trust status with the Trust Registry before accepting the credential into storage — mirroring the OID4VCI on-device issuance pattern this repo implements (`src/services/vci/**`).
