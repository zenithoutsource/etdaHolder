# Developer Onboarding — Minimal Env & Setup Script

**Date:** 2026-07-08  
**Status:** Approved (2026-07-08)  
**Audience:** New developers on this repo (first slice of broader “easy to reuse” goal)

## Problem

New developers face a high-friction first run:

- Root `.env.example` mixes **required production settings**, **dev-only VPN proxies**, **verifier compatibility probes**, **ETDA-specific issuer portal URLs**, and **NFC/dual-format tuning** (~25 variables).
- `server/.env.example` adds another layer (DB, SMTP, dev proxies, VP relay, mDOC issuer).
- Config is read directly from `process.env` across many files under `src/config/` and `src/sdk/` with no schema or profiles.
- `installWalletApiFetch.ts` defaults to `http://localhost:3001` while the local backend listens on **4000** — a silent mismatch for anyone who skips env setup.

The goal is not full SDK extraction yet. The goal is: **clone → run on physical Android → wallet home loads in ~30 minutes**, with minimal env and one setup command.

## Success Criteria

A new developer on **Windows + XAMPP + physical Android** can:

1. `yarn install` → `yarn setup`
2. Start XAMPP MySQL and apply migrations
3. `cd server && yarn install && yarn dev` (port 4000)
4. `yarn android:dev`
5. Register/login → **Wallet home loads** (empty credentials OK) — no crash

**Explicitly out of scope for this slice:**

- OID4VCI QR issuance, OID4VP presentation, NFC, VPN issuer/verifier proxies
- Certificate pinning, push notification registration
- SDK packaging / monorepo extraction (audiences B & C — follow-up work)

## Chosen Approach

**Approach 1 (onboarding-only)** with a thin validation slice from Approach 2:

- Split env into **required** vs **optional**
- Add `yarn setup` to generate day-one files
- Single **Getting Started** doc as the entry point
- Fix backend URL default mismatch (3001 → 4000)
- **No** service-layer refactor or monorepo split in this slice

Rejected for now:

- **Full centralized config module** — touches too many files for v1
- **Full SDK monorepo** — weeks of work; premature for audience A

## Environment File Layout

| File | Purpose | When needed |
|------|---------|-------------|
| `.env` | Required — generated by `yarn setup`, 2 vars | Day 1 |
| `.env.development.local` | Optional dev overrides (gitignored) | Advanced / ETDA office |
| `.env.development.local.example` | All optional mobile vars (migrated from current bloated `.env.example`) | Reference only |
| `server/.env` | Required — generated by `yarn setup`, 8 vars | Day 1 |
| `server/.env.development.local.example` | Optional server hacks (proxies, VP relay, mDOC, SMTP) | Reference only |

Root `.env.example` becomes a **pointer file** (~10 lines) linking to `yarn setup` and `.env.development.local.example`.

### Minimal mobile `.env` (generated)

```env
EXPO_PUBLIC_WALLET_API_BASE_URL=http://<lan-ip>:4000
EXPO_PUBLIC_SKIP_PUSH_REGISTRATION=true
```

All other mobile vars (`EXPO_PUBLIC_ENABLE_WALLET_DEBUG_LOGS`, `EXPO_PUBLIC_WALLET_KEY_*`, PIN grace, expiry window, verifier probes, issuer portals, NFC/dual-format) keep **existing code defaults** when unset — no need to copy on day one. (`EXPO_PUBLIC_ENABLE_WALLET_DEBUG_LOGS` is enabled in `__DEV__` when unset per `walletLogger.ts`; document in optional example file only.)

### Minimal `server/.env` (generated)

```env
PORT=4000
DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=etda_wallet
DB_USER=root
DB_PASSWORD=
JWT_SECRET=<random-dev-secret>
JWT_EXPIRES_IN=7d
```

SMTP, dev issuer/verifier proxies, VP relay, mDOC issuer → optional example file only.

## `yarn setup` Script

**Location:** `scripts/setup-dev-env.js`  
**Package script:** `"setup": "node scripts/setup-dev-env.js"`

### Behavior

1. Detect LAN IP via a **new** helper using `os.networkInterfaces()` — skip virtual/VPN/tunnel adapters (e.g. names matching `Virtual`, `VPN`, `TAP`, `WSL`, `Hyper-V`, `Loopback`). There is **no** existing IP-detection code in `scripts/`; `run-android-device.js` only passes `--host lan` to Expo and must not be cited as a reuse source.
2. Prompt to confirm detected IP or enter manually.
   - **`--yes` / CI:** skip the prompt; use the best detected candidate.
   - **If no candidate is found** (VPN-only or offline): write `http://localhost:4000`, print a warning that physical-device testing may need a manual LAN IP or `adb reverse`, and exit **`0`** (setup still produces usable files). Do not fail silently without the warning.
   - **If `--yes` and setup cannot write env files** (e.g. prerequisite missing): exit **`3`**.
3. Write root `.env` if missing; if it already exists, **skip** and print a notice (suggest `--force` to regenerate).
4. Write `server/.env` if missing; same skip-with-notice behavior.
5. Print a short **next steps** checklist (XAMPP → migrations → server → android:dev).
6. **`yarn setup --check`** (read-only): verify `.env` and `server/.env` exist, optional MySQL TCP probe, optional HTTP health on port 4000. Exit **`0`** when all requested checks pass; exit **`1`** when any check fails (missing env file, MySQL unreachable, server not responding).

### Safety

- **Never overwrite** existing `.env` or `server/.env` without `--force`.
- **Idempotent reruns:** existing env files are skipped with a notice; setup still exits **`0`** (not an error).
- No committed secrets.

### Code fix (same PR)

Change `DEFAULT_WALLET_API_BASE_URL` in `src/sdk/installWalletApiFetch.ts` from `http://localhost:3001` to `http://localhost:4000`.

**Side benefit:** `resolveNativeDevLoopbackBaseUrl` already rewrites `localhost` → Metro dev-server host on physical devices in `__DEV__`. Aligning the default to port **4000** gives a working **zero-env** dev path when a developer skips `yarn setup` but runs the local backend on the documented port.

## Error Handling

- **`yarn setup`:** `0` success (wrote new files and/or skipped existing with notice; includes `--yes` localhost fallback + warning), `1` user abort, `3` unrecoverable `--yes` failure (e.g. cannot write target directory)
- **`yarn setup --check`:** `0` all checks pass, `1` any check failed
- Getting Started troubleshooting: map common errors to fix (wrong IP, XAMPP not running, wrong port)

## Security Notes

- Generated `JWT_SECRET` is for **local dev only**; documented in Getting Started
- Optional vars that weaken security (`EXPO_PUBLIC_DISABLE_SD_JWT_KB_FOR_TESTING`, etc.) live only in `.env.development.local.example` with bold “never for release” warnings
- No change to production runtime policy (`assertWalletApiRuntimePolicy` still requires HTTPS + pins in release builds)
