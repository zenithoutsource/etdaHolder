Approved rev 4 Android HCE dual-format อัปเดต 2026-07-06

อธิบายแบบเห็นภาพ: Android HCE Dual-Format Presentation

หน้านี้สรุปสเปก android-hce-dual-format-presentation-design.md rev 4 เป็นภาษาไทย เพื่อให้เข้าใจภาพรวมของ dual-format credential: dc+sd-jwt สำหรับ JSON/SD-JWT VC และ mso_mdoc สำหรับ ISO 18013-5 mDOC. Offline NFC ใช้ full ISO 18013-5 NFC retrieval ผ่าน Android HCE, ไม่ใช้ BLE data leg บนมือถือ

สิ่งที่เปลี่ยนจากสเปกเดิม

เลิก BLE data leg บนมือถือ

สเปกเดิมใช้ NFC device engagement แล้วส่งข้อมูลผ่าน BLE. Rev 4 เปลี่ยนเป็น ISO 7816-4 APDUs ผ่าน Android HCE เต็มรูปแบบ

ACR1311U-N2 ยังใช้ Bluetooth

Bluetooth เป็นลิงก์ระหว่าง reader กับ host ของ reader เท่านั้น Wallet มองเห็นเป็น NFC/HCE ไม่ต้องเปิด BLE server ในมือถือ

โทรศัพท์ไม่ล็อกยี่ห้อ

ข้อกำหนดคือ Android 10+ ที่รองรับ NFC และ HCE. เครื่องจริงที่มีอยู่คู่กับ ACR1311U-N2 เป็น acceptance target แรก

ภาพรวมในหนึ่งหน้า

Wallet เก็บ 2 รูปแบบ

เอกสารเดียวกันจะมี credential สอง format แยกกัน คือ SD-JWT VC และ mDOC แต่ผูกเป็นเอกสารเดียวด้วย logicalCredentialId

Online ใช้ OpenID4VP

Verifier สามารถขอทั้ง mso_mdoc และ dc+sd-jwt ใน transaction เดียว Wallet ต้องส่งทั้งคู่ถ้าถูกขอแบบ required

Offline NFC ใช้ Android HCE

การแตะเครื่องอ่านใช้ full ISO 18013-5 NFC data retrieval: SELECT AID, ENVELOPE chaining, GET RESPONSE และ encrypted SessionData ผ่าน Android HCE

ยืนยันก่อนแตะ

ผู้ใช้ต้องเห็น disclosure set จาก fixed reader profile แล้วกดอนุญาตก่อนเปิด HCE. ถ้า reader ขอเกินที่อนุญาต Wallet ต้องปฏิเสธ

จุดสำคัญ: JSON companion ต้องเป็น SD-JWT VC presentation ที่มี KB-JWT ผูกกับ nonce จาก reader ห้ามส่งเป็น JSON ธรรมดาที่ตรวจสอบลายเซ็นไม่ได้ และต้องอยู่ใต้ proprietary companion AID แยกจาก mDOC AID

ขอบเขตมาตรฐาน

สิ่งที่เป็นมาตรฐาน

  • OID4VCI 1.0 สำหรับออก credential
  • OpenID4VP 1.0 สำหรับ online presentation
  • ISO 18013-5 DeviceResponse สำหรับ mDOC offline NFC
  • APDU ผ่าน Android HCE สำหรับ proximity exchange
  • ISO 7816-4 SELECT AID / ENVELOPE / GET RESPONSE สำหรับ NFC retrieval

สิ่งที่เป็น proprietary extension

  • JSON companion payload ใน reader session เดียวกัน
  • proprietary companion AID แยกจาก mDOC AID
  • คำสั่ง APDU เฉพาะของ companion เช่น capabilities และ nonce request
  • โหมด dual-format ที่ reader ต้อง negotiate ชัดเจน
  • fixed reader profile สำหรับ pre-tap consent ใน v1
Generic ISO 18013-5 reader ต้องยังอ่าน mDOC ได้ตามปกติ แม้ไม่รู้จัก companion JSON companion เลย. Reader ที่ไม่เลือก companion AID จะได้ mDOC-only behavior

การออก Credential แบบ Dual-Format

Issuer ควรเสนอ credential configuration สองตัวสำหรับเอกสารเดียวกัน เช่น TranscriptCredential_dc+sd-jwt และ TranscriptCredential_mso_mdoc Wallet ต้องยิง Credential Endpoint แยกกัน เพราะต่าง format ถือเป็น credential request คนละรายการ. QR หรือ credential_offer_uri ไม่มี holder claims จริง เช่น ชื่อหรือข้อมูล transcript; Issuer resolve claims จาก server-side state

1Resolve offer
อ่าน QR หรือ credential offer URI
2Fetch metadata
ดึงข้อมูล Issuer และ configuration
3Group configs
จับคู่ family suffix _dc+sd-jwt / _mso_mdoc หรือใช้ explicit grouping field
4Token exchange
แลก pre-authorized code เป็น access token
5Request both
ขอ dc+sd-jwt และ mso_mdoc แยก request
6Link and validate
ผูกด้วย logicalCredentialId แล้วตรวจ consistency
การ claim dual-format เป็น user action เดียว จึงต้องมี biometric/device-auth ไม่เกินหนึ่งครั้ง แม้จะต้อง sign proof สองครั้ง. ถ้า format หนึ่ง fail ให้แสดง partial issuance และ retry เฉพาะเมื่อ token ยัง valid

Logical Credential

logicalCredentialId
ผูกเอกสารเดียวกัน
documentType / issuer / subjectId / documentId

Physical Format Records

dc+sd-jwt record
+
mso_mdoc record
ใช้ key อะไรผูกกัน

ดีที่สุดคือ Issuer ส่ง stable logical ID เดียวกันในทั้งสอง format ถ้าไม่มี ให้ derive จาก issuer, document type, subject ID และ document ID

ห้ามใช้ full claim hash

เพราะ SD-JWT selective disclosure, mDOC namespace, encoding และชื่อ claim อาจต่างกัน ทำให้ hash ไม่เสถียร

สถานะ consistency

verified, warning, หรือ mismatch เพื่อบอกว่า credential สองรูปแบบอ้างถึงเอกสารเดียวกันได้มั่นใจแค่ไหน

Migration v1

LogicalCredential เป็น linking layer ทับ VerifiableCredentialRecord เดิม. Home, expiry, renewal, lifecycle และ backend sync ยัง key off SD-JWT record id

Renewal v1 ยัง SD-JWT-only

การ renew mDOC และ paired-refresh policy เป็น slice ถัดไป ตอนนี้ renewal ใช้ flow เดิมของ SD-JWT credential

Online Presentation: OpenID4VP

ถ้า DCQL request ขอทั้ง mso_mdoc และ dc+sd-jwt Wallet ต้อง match credential ที่มี format ครบและใช้ logicalCredentialId เดียวกัน

1Resolve request
อ่าน Verifier request
2Match formats
หาเอกสารที่มี format ตามที่ขอ
3One consent screen
แสดง mDOC fields และ JSON fields รวมในหน้าขออนุญาตเดียว
4Sign and submit
ส่งทั้งสอง presentation ใน vp_token
ถ้า Verifier ระบุว่าทั้งสอง format เป็น required แต่ Wallet มีไม่ครบ ต้อง fail request ห้ามส่งแค่ format ที่มีแบบเงียบๆ

Holder Binding และ Key Decision

ใช้ EdDSA/Ed25519 เป็นค่าเริ่มต้น

mDOC device key ใช้ Keychain-protected Ed25519 seed เดียวกับ Holder identity key เพื่อให้ SD-JWT และ mDOC มี key lifecycle เดียวกัน

ต้องผ่าน interop ก่อน production

Issuer และ ACR1311U-N2 verifier stack ต้องยืนยันว่า Ed25519 deviceSignature ใน DeviceResponse ตรวจสอบได้จริง

Fallback ถ้า EdDSA ไม่ผ่าน

ถ้า reader หรือ Issuer ไม่รับ EdDSA device authentication ให้เปลี่ยนเฉพาะ mDOC device key เป็น P-256 AndroidKeyStore และบันทึกใน holderBindingRef

Offline NFC/HCE: ลำดับที่ถูกต้อง

NFC reader timeout เร็วมาก จึงไม่ควรให้ผู้ใช้อ่านรายละเอียดและกดอนุญาตระหว่าง APDU session การออกแบบนี้กำหนดให้ consent เกิดก่อนแตะ reader โดย v1 ใช้ fixed reader profile เป็น disclosure set ที่ผู้ใช้เห็นก่อนแตะ

1Review profile
ผู้ใช้เปิด presentation screen ดู mDOC-only หรือ dual-format และ fixed profile field list
2Approve
อนุญาตหนึ่งครั้ง และทำ signing/authentication ที่จำเป็น
3Arm HCE
เปิด HCE ในกรอบเวลา เช่น 60 วินาที
4Tap reader
reader เลือก mDOC AID แล้วส่ง SessionEstablishment
5Check request
decrypt DeviceRequest แล้วตรวจว่าไม่เกิน approved profile
6Send payloads
ส่ง encrypted DeviceResponse และถ้า reader เลือก companion AID จึงส่ง SD-JWT companion
mDOC-only: จบหลัง DeviceResponse dual-format: mDOC + SD-JWT companion unarmed tap: ไม่ตอบ AID และไม่แชร์ข้อมูล
Approval เป็นเพดาน ไม่ใช่ขั้นต่ำ: ถ้า reader ขอ subset ของ profile ให้ส่งเฉพาะที่ขอได้ แต่ถ้าขอ field นอก profile ต้องคืน ISO 18013-5 error status และจบ session

ISO 18013-5 Session Layer

Device engagement

เริ่มจาก QR engagement หรือ NFC negotiated handover พร้อม ephemeral public key ของ Wallet (EDeviceKey)

Session establishment

reader ส่ง SessionEstablishment พร้อม EReaderKey จากนั้นทั้งสองฝั่ง derive session keys ด้วย ECDH P-256 + HKDF

Encrypted SessionData

request และ response หลังจากนั้นต้องเข้ารหัสทั้งหมด ไม่ส่ง DeviceRequest หรือ DeviceResponse แบบ plaintext

APDU transport

ใช้ SELECT AID, ENVELOPE chaining และ GET RESPONSE ตาม ISO 7816-4 บน Android HCE

companion JSON Companion

วิธีส่ง

  • reader ต้อง select proprietary companion AID หลัง mDOC session
  • reader ต้องส่ง GET CAPABILITIES เพื่อ negotiate dual-format mode
  • payload ใช้ APDU chaining เช่นเดียวกับ mDOC

payload คืออะไร

  • SD-JWT VC presentation
  • selected disclosures เฉพาะที่อนุมัติ
  • KB-JWT ที่ bind กับ nonce จาก reader
  • ไม่ใช่ unsigned JSON และไม่ใช่ส่วนหนึ่งของ ISO 18013-5
ก่อน implement ต้องมี companion-extension spec ที่ pin CLA/INS, capability format และ nonce format ให้ชัดเจน. Companion ต้องไม่ alter mDOC exchange; reader ต้องเลือก companion AID หลัง ISO 18013-5 session จบ

HCE State Machine

idle
ยังไม่เริ่ม session
awaiting-consent
ผู้ใช้เปิดหน้าพresentation และกำลังตรวจ disclosure set
approved
ผู้ใช้อนุญาตแล้ว ทำ signing/authentication ที่จำเป็นในจุดนี้
hce-armed
เปิด HCE service พร้อม arm-window timer และข้อมูลที่อนุมัติไว้
reader-selected
reader เลือก Wallet mDOC AID
session-established
derive session keys และตรวจ session encryption สำเร็จ
request-received
ถอดรหัส DeviceRequest ตรวจว่าไม่เกิน disclosure set ที่อนุมัติ; subset ส่งได้, out-of-profile ต้อง error
transmitting-mdoc
สร้างและส่ง encrypted mDOC DeviceResponse
transmitting-json-companion
reader เลือก companion AID แล้วรับ SD-JWT companion
complete / cancelled / error
จบสำเร็จ ยกเลิก หรือเกิด protocol/reader/signing failure แล้วล้าง buffers

Security และ Privacy Gates

Screen-on required

Android HCE service ต้องตั้ง android:requireDeviceScreenOn="true" และไม่ตอบเมื่อหน้าจอดับ

One-prompt rule

หนึ่ง user action ต้องมี biometric/device-auth ไม่เกินหนึ่งครั้ง ถ้า signing gate ทำ auth แล้ว ห้ามซ้อน prompt แยก

Ed25519 holder/device key

ใช้ EdDSA/Ed25519 seed เดียวกับ holder identity ปัจจุบัน แต่ต้อง validate กับ Issuer และ ACR1311U-N2 ก่อน; fallback คือ P-256 AndroidKeyStore เฉพาะ mDOC

Payload size cap

ถ้ารวม mDOC และ companion ใหญ่เกิน EXPO_PUBLIC_NFC_PAYLOAD_MAX_BYTES ต้อง fail ก่อนแตะ ไม่ใช่ล้มกลาง transfer

Logs ห้ามมี raw VC, VP, SD-JWT, mDOC CBOR, DeviceResponse, APDU payload, tokens, private keys, seeds, holder claims หรือ PII

ค่าที่ต้อง configurable

EXPO_PUBLIC_DUAL_FORMAT_ISSUE_SKEW_MS
default 300000 ms: ความต่าง issued-at สูงสุดระหว่างสอง format ก่อนขึ้น warning
EXPO_PUBLIC_HCE_ARM_WINDOW_MS
default 60000 ms: เวลาที่ HCE ถูก arm หลังผู้ใช้ approve
EXPO_PUBLIC_NFC_PAYLOAD_MAX_BYTES
default 65536 bytes: hard cap ของ payload ที่ส่งผ่าน NFC; target รวม mDOC + companion ไม่เกิน 32 KB

Implementation Stack

Native HCE service

ต้องใช้ Kotlin HostApduService ผ่าน Expo config plugin/local module. ไม่มี pure-JS HCE ที่พอสำหรับ production

mDOC engine

ควรใช้ audited library. Multipaz เป็น leading candidate แต่ยังไม่ final จนกว่าจะผ่าน physical-device interop และ ADR ถัดไป

JS ↔ native boundary

JS ดูแล consent, credential selection, state display. Native ดูแล APDU, session crypto, buffers และ clear payload หลัง complete/cancel/error

Acceptance Criteria แบบเข้าใจง่าย

Issuance
Wallet claim ได้ทั้ง SD-JWT VC และ mDOC จาก offer เดียว แล้วผูกเป็น logical credential เดียว
EdDSA interop
reader/verifier stack ตรวจ Ed25519 deviceSignature ใน DeviceResponse ได้จริงก่อน lock key decision
Consistency
Wallet ตรวจ mismatch/warning ได้ เช่น issuer, subject, document ID, issued-at หรือ expiry ไม่ตรงกัน
Online VP
OpenID4VP ส่งทั้งสอง format ใน vp_token ได้เมื่อ Verifier ขอ; current path ต้องเพิ่ม multi-format assembly
Offline HCE
ACR1311U-N2 ทำ full ISO 18013-5 session สำเร็จ: engagement, session encryption, encrypted DeviceResponse และ companion companion
Consent
ผู้ใช้เห็น consent screen เดียวก่อนแตะ และ unarmed tap ต้องไม่แชร์ข้อมูล
Failure safety
disconnect, timeout, screen off, over-size payload และ unsupported request ต้องไม่ทำให้แชร์ข้อมูลเงียบๆ

สิ่งที่ยังอยู่นอก scope

  • iOS HCE
  • Generic BLE verifier app ที่ไม่เกี่ยวกับ ACR1311U-N2
  • NFC NDEF issuance หรือ tag reading
  • ระบบ reader proprietary ที่ไม่ใช่ companion
  • การ redesign UI ราย claim ที่ไม่เกี่ยวกับ dual-format consent
  • Variable reader templates และ reader-engagement QR pre-tap request flow
