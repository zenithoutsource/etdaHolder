ETDA Digital Wallet — Tech Overview

OID4VCI 1.0 Holder Wallet  ·  Expo SDK 54  ·  iOS & Android

What It Is

Mobile wallet for Thai citizens. Receives government-issued Verifiable Credentials (national ID, driving licence, transcripts) via the OID4VCI 1.0 open standard, stores them encrypted on-device, and presents them to verifiers via NFC (ISO 18013-5) or QR code. Private keys never leave the device’s hardware secure enclave.

How It Works

1
Receive Credential Offer

User scans a QR code, taps an NFC tag, or taps “Claim” inside the app. This produces an openid-credential-offer:// URI.

On-Device
2
Resolve Issuer Metadata

@sphereon/oid4vci-client fetches /.well-known/openid-credential-issuer from the government Issuer (ThaID, DLT, etc.) to discover supported credential types and endpoints.

Network → Issuer
3
Token Exchange

Pre-authorized code flow: exchanges the offer code at the Issuer’s token endpoint and receives an access token + c_nonce.

Network → Issuer
4
Proof of Possession (PoP) JWT

App calls signProof() → native module uses the hardware key (etda_wallet_signing_key) inside Secure Enclave / Android Keystore to sign the JWT. Biometric fires here. Private key bytes never reach JavaScript.

Hardware Key
5
Claim Credential from Issuer

PoP JWT is sent in the credential request. The Issuer returns a signed VC JWT. App validates the JWT structure on-device.

Network → Issuer
6
Store Locally (Encrypted)

VC saved to react-native-mmkv with AES-256 encryption. Encryption key lives in the hardware keychain, unlocked by biometric on first use.

On-Device
7
Sync to Company Backend

App calls POST /wallet-api/wallet/{walletId}/credentials/import via the Orval-generated TanStack Query SDK. This is the only backend call in the entire flow. The backend never sees OID4VCI traffic.

Network → Company API
8
Present Credential

User taps phone to a verifier’s NFC reader. ISO 18013-5 proximity mdoc exchange transfers only the requested claims. iOS uses BLE engagement fallback.

Tech Stack

Expo SDK 54 + Hermes
Runtime. Hermes engine required — all crypto on signing path is native JSI, not pure-JS.
@animo-id/
expo-secure-environment
Generates & uses the hardware EC P-256 key. Signs PoP JWT inside Secure Enclave / Keystore.
react-native-mmkv
AES-256 encrypted key-value store. Holds all credentials on-device.
react-native-quick-crypto
Non-signing crypto: hashing, HMAC, base64url. Hermes-safe (no BigInt).
@sphereon/
oid4vci-client
Runs the full OID4VCI 1.0 protocol on-device: offer → metadata → token → credential.
orval + TanStack Query
Auto-generates TypeScript hooks from company OpenAPI spec. All backend calls go through this.
zustand
Global state — credentials slice + session slice. Persisted via MMKV adapter.
expo-router
File-system-based navigation (screens map to files in app/).
NativeWind
Tailwind CSS utility classes for React Native UI. Config-driven card rendering.

Security Rules

🔐
Private key never in JavaScript

Key lives in hardware only. JS calls the native module by alias — never touches key bytes.

👆
Biometric on every sign

Hardware-enforced. No software bypass. Cancelling biometric aborts credential acquisition.

🚫
AsyncStorage is banned

Blocked at lint + CI level. All credential data goes through encrypted MMKV only.

🏭
Backend never proxies OID4VCI

Company server authenticates the user and hands back an offer URL — nothing more. Credential issuance is direct device ↔ Issuer.

📋
Self-sovereign Holder DID

did:key derived from the device’s P-256 public key. No server needed to resolve it.
