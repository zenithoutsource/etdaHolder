กระเป๋าเงินดิจิทัลบนมือถือ ให้ประชาชนไทย รับ เก็บ และแสดง เอกสารราชการในรูปแบบดิจิทัล เช่น บัตรประชาชน ใบขับขี่ ทรานสคริปต์ ทำงานตามมาตรฐานเปิด OID4VCI 1.0 (รับเอกสาร) และ OID4VP 1.0 (แสดงเอกสาร) กุญแจส่วนตัวไม่เคยออกจากชิปความปลอดภัยของอุปกรณ์
Mobile wallet — receives government VCs via OID4VCI 1.0, stores AES-256 on-device, presents to verifiers via OID4VP 1.0 and ISO 18013-5 NFC proximity exchange. Private keys are hardware-bound and non-extractable.

สองมาตรฐานหลัก

📥 OID4VCI 1.0 — รับเอกสาร (Issuance)

โปรโตคอลสำหรับ รับ Verifiable Credential จากผู้ออกเอกสาร (ThaID, กรมการขนส่ง, มหาวิทยาลัย)

  • รับ Credential Offer ผ่าน QR / NFC / In-app
  • ดึง Metadata จาก Issuer (/.well-known)
  • แลก Token — Pre-authorized Code Flow
  • กรอก tx_code ถ้า Issuer ส่ง PIN มา
  • ส่ง PoP JWT พร้อมลายเซ็น hardware
  • รับ VC JWT หรือ SD-JWT VC กลับมา
📤 OID4VP 1.0 — แสดงเอกสาร (Presentation)

โปรโตคอลสำหรับ แสดง Verifiable Credential ให้ผู้ตรวจสอบ (ธนาคาร, สนามบิน, ห้างฯ)

  • รับ Authorization Request จาก Verifier (QR / deeplink)
  • อ่าน Presentation Definition ว่าต้องการ claim อะไร
  • ผู้ใช้ยืนยันและเลือก claim (Selective Disclosure)
  • สร้างและเซ็น VP Token ด้วย hardware key
  • ส่ง vp_token + presentation_submission กลับ
  • NFC: ISO 18013-5 proximity mdoc exchange

ขั้นตอน: รับเอกสาร (OID4VCI 1.0)

Credential Issuance Flow
1
รับ Credential Offer
สแกน QR code แตะ NFC หรือกด "รับเอกสาร" ในแอป — ได้ openid-credential-offer:// URI มา
บนอุปกรณ์
2
ดึง Issuer Metadata
@sphereon/oid4vci-client ดึง /.well-known/openid-credential-issuer — รู้ว่า endpoint อยู่ที่ไหน รองรับ credential format อะไร
Network → Issuer
3
แลก Token
ส่ง pre-authorized code ไปที่ token endpoint — ได้ access token + c_nonce กลับมา
Primary: pre-authorized code flow  |  Planned Phase 4: authorization code flow
Network → Issuer
4
กรอก tx_code (ถ้ามี)
ถ้า Issuer ส่ง PIN มาทาง SMS/อีเมล — tx_code ใน Credential Offer จะบอกว่าต้องกรอกหรือไม่ และเป็นตัวเลขหรือตัวอักษร ผู้ใช้กรอกก่อน token exchange
User Input
5
สร้าง Proof of Possession (PoP) JWT
เรียก signProof() → native module เซ็น JWT ด้วย etda_wallet_signing_key ภายใน Secure Enclave / Keystore — ต้องผ่าน biometric ทุกครั้ง private key bytes ไม่เคยถึง JavaScript
Hardware Key
6
ขอรับ Credential จาก Issuer
ส่ง PoP JWT ไปกับ credential request — ได้ VC JWT หรือ SD-JWT VC กลับมา แอปตรวจสอบโครงสร้างบนอุปกรณ์
Network → Issuer
7
เก็บลงอุปกรณ์ (AES-256)
VC เข้ารหัสแล้วเก็บใน react-native-mmkv — กุญแจเข้ารหัสอยู่ใน hardware keychain ต้องผ่าน biometric ครั้งแรก
บนอุปกรณ์
8
ซิงก์ไป Company Backend
เรียก POST /wallet-api/wallet/{walletId}/credentials/import ผ่าน Orval SDK — เรียก backend เพียงครั้งเดียวตลอด flow ทั้งหมด backend ไม่เคยเห็น OID4VCI traffic
Network → Company API

ขั้นตอน: แสดงเอกสาร (OID4VP 1.0)

Credential Presentation Flow
1
รับ Authorization Request จาก Verifier
สแกน QR code หรือรับ deeplink จากผู้ตรวจสอบ — Request ระบุ presentation_definition บอกว่าต้องการเอกสารอะไร
บนอุปกรณ์
2
จับคู่ Credential กับ Presentation Definition
แอปหาว่าเอกสารใน wallet ตรงกับที่ Verifier ขอหรือไม่ โดยใช้ DIF Presentation Exchange
บนอุปกรณ์
3
ผู้ใช้ยืนยันและเลือก Claim (Selective Disclosure)
แอปแสดงว่าจะเปิดเผยข้อมูลอะไรบ้าง ผู้ใช้กดยืนยัน — SD-JWT VC ให้เลือกได้ว่าจะส่ง claim ไหน เช่น แสดงแค่ "อายุ ≥ 20" โดยไม่เปิดวันเกิดจริง
User Consent
4
สร้างและเซ็น VP Token
สร้าง Verifiable Presentation รวม credential ที่เลือก แล้วเซ็นด้วย hardware key — ต้องผ่าน biometric อีกครั้ง
Hardware Key
5
ส่ง Presentation Submission กลับ Verifier
ส่ง vp_token + presentation_submission ผ่าน response_mode=direct_post หรือผ่าน NFC
Network → Verifier
6
NFC Proximity — ISO 18013-5
กรณีแตะมือถือกับเครื่องอ่าน — ใช้ mdoc proximity exchange ถ่ายโอนเฉพาะ claim ที่ขอ iOS ใช้ BLE engagement เป็น fallback
NFC / Hardware

กฎความปลอดภัยหลัก

🔐
Private key ไม่เคยอยู่ใน JavaScript

Key อยู่แค่ใน hardware chip — JS เรียกผ่าน alias เท่านั้น ไม่เคยแตะ key bytes โดยตรง

👆
Biometric ทุกครั้งที่เซ็น

บังคับระดับ hardware ทั้งตอนรับและแสดงเอกสาร — ไม่มี software bypass ยกเลิก biometric = ยกเลิก operation

🚫
ห้ามใช้ AsyncStorage

บล็อกระดับ lint + CI — ข้อมูลทั้งหมดต้องผ่าน MMKV ที่เข้ารหัสเท่านั้น

🏭
Backend ไม่เคย proxy OID4VCI/OID4VP

Company server แค่ยืนยันตัวตนและส่ง offer URL กลับมา — issuance และ presentation ทำตรงระหว่างอุปกรณ์กับ Issuer/Verifier

📋
Holder DID เป็นของตัวเอง (Self-Sovereign)

did:key derive จาก P-256 public key ของอุปกรณ์ — ไม่ต้องพึ่ง server ใดในการ resolve

👁
Selective Disclosure — เปิดเผยเฉพาะที่จำเป็น

SD-JWT VC ให้เลือกได้ว่าจะเปิด claim ไหน เช่น แสดงแค่ "อายุ ≥ 20" โดยไม่เปิดวันเกิดจริง ทำได้ผ่าน @sd-jwt/core
