แอปนี้คืออะไร

กระเป๋าเงินดิจิทัลบนมือถือ ให้ประชาชนไทย รับ เก็บ และแสดง เอกสารราชการในรูปแบบดิจิทัล เช่น บัตรประชาชน (ThaID), ใบขับขี่ (DLT), ทรานสคริปต์มหาวิทยาลัย ทำงานตามมาตรฐานเปิดสากล OID4VCI 1.0 (รับเอกสาร) และ OID4VP 1.0 (แสดงเอกสาร) กุญแจส่วนตัวไม่เคยออกจากชิปความปลอดภัยของอุปกรณ์ (Secure Enclave / Android Keystore) ผู้ใช้ถือ credential ของตัวเอง — ไม่ขึ้นอยู่กับ server กลางใดทั้งสิ้น
Mobile wallet for Thai citizens: acquires government VCs via OID4VCI 1.0, stores AES-256 encrypted on-device, presents to verifiers via OID4VP 1.0 and ISO 18013-5 NFC proximity exchange. All signing is hardware-bound with biometric gate. Self-sovereign — no central server dependency for key operations.

สองมาตรฐานหลักที่ใช้

📥 OID4VCI 1.0 — รับเอกสาร (Credential Issuance)

โปรโตคอลมาตรฐานสากลสำหรับ รับ Verifiable Credential จากหน่วยงานรัฐ

  • รับ Credential Offer ผ่าน QR / NFC / In-app
  • ดึง Issuer Metadata จาก /.well-known
  • Token Exchange (Pre-authorized Code Flow)
  • กรอก tx_code ถ้า Issuer ส่ง PIN มา
  • ส่ง Proof of Possession (PoP) JWT
  • รับ VC JWT หรือ SD-JWT VC
  • ตรวจสอบ Issuer signature ผ่าน did:web
📤 OID4VP 1.0 — แสดงเอกสาร (Credential Presentation)

โปรโตคอลมาตรฐานสากลสำหรับ แสดง Verifiable Credential ให้ผู้ตรวจสอบ

  • รับ Authorization Request (QR / deeplink / NFC)
  • อ่าน Presentation Definition (DIF)
  • ผู้ใช้เลือก Claim ที่จะเปิดเผย (Selective Disclosure)
  • สร้างและเซ็น VP Token ด้วย hardware key
  • ส่ง vp_token + presentation_submission
  • NFC: ISO 18013-5 proximity mdoc exchange

ระบบ DID (Decentralized Identifier)

👤 Holder — ผู้ถือเอกสาร

did:key
  • สร้างจาก P-256 public key ของอุปกรณ์
  • Self-contained — ไม่ต้อง server ในการ resolve
  • ปรากฏใน PoP JWT field iss และ VC field sub
  • รูปแบบ: did:key:z<base58btc(multicodec + key)>
  • เปลี่ยนไม่ได้หลัง generate — ผูกกับอุปกรณ์

🏭 Issuer — หน่วยงานออกเอกสาร

did:web
  • ปรากฏใน VC field iss
  • Wallet resolve โดยดึง DID document จาก:
  • https://<domain>/.well-known/did.json
  • ใช้ public key จาก DID document ตรวจลายเซ็น VC
  • ตัวอย่าง: did:web:issuer.thaid.go.th

ขั้นตอนละเอียด: รับเอกสาร

OID4VCI 1.0 — Credential Issuance Flow
1
รับ Credential Offer
สแกน QR code ผ่านกล้อง, แตะ NFC tag, หรือกด "รับเอกสาร" ในแอป — ได้ openid-credential-offer:// URI มา
Library: expo-camera / react-native-vision-camera · react-native-nfc-manager
บนอุปกรณ์
2
ดึง Issuer Metadata
@sphereon/oid4vci-client ดึง /.well-known/openid-credential-issuer จาก Issuer — รู้ endpoint, credential format ที่รองรับ, display metadata (ชื่อ สี โลโก้)
Network → Issuer
3
แลก Token
ส่ง pre-authorized code ไปที่ token endpoint — ได้ access token + c_nonce กลับมา
Primary: pre-authorized code flow · Phase 4: authorization code flow
Network → Issuer
4
กรอก tx_code (ถ้ามี)
ถ้า Credential Offer มี tx_code — Issuer ส่ง PIN มาทาง SMS หรืออีเมล ผู้ใช้กรอกก่อน ระบุด้วย input_mode (numeric/text) และ length
User Input
5
สร้าง Proof of Possession (PoP) JWT
เรียก signProof() → native module เซ็น JWT ด้วย etda_wallet_signing_key ภายใน Secure Enclave / Keystore — ต้องผ่าน biometric ทุกครั้ง private key bytes ไม่เคยถึง JavaScript
Hardware Key
6
ขอรับ Credential
ส่ง PoP JWT ในรูป proof field ของ credential request — Issuer ส่ง VC JWT หรือ SD-JWT VC กลับมา
Network → Issuer
7
ตรวจสอบ Issuer Signature (did:web)
Wallet resolve did:web ของ Issuer → ดึง DID document จาก https://<issuer-domain>/.well-known/did.json → ใช้ public key ตรวจลายเซ็น VC บนอุปกรณ์
บนอุปกรณ์
8
เก็บลงอุปกรณ์ (AES-256)
VC เข้ารหัสแล้วเก็บใน react-native-mmkv — กุญแจเข้ารหัสอยู่ใน hardware keychain ต้องผ่าน biometric ครั้งแรก
บนอุปกรณ์
9
ซิงก์ไป Company Backend
เรียก POST /wallet-api/wallet/{walletId}/credentials/import ผ่าน Orval-generated SDK — เรียก backend เพียงครั้งเดียวตลอด flow ทั้งหมด backend ไม่เคยเห็น OID4VCI traffic
Network → Company API

ขั้นตอนละเอียด: แสดงเอกสาร

OID4VP 1.0 — Credential Presentation Flow
1
รับ Authorization Request จาก Verifier
สแกน QR code จากเครื่อง Verifier หรือรับ deeplink — Request มี presentation_definition บอกว่าต้องการ credential ประเภทไหนและ claim อะไร
บนอุปกรณ์
2
จับคู่ Credential กับ Presentation Definition (DIF)
แอปใช้ DIF Presentation Exchange หาว่าเอกสารใน wallet ตรงกับที่ Verifier ขอหรือไม่ ถ้าตรงหลายใบให้ผู้ใช้เลือก
บนอุปกรณ์
3
ผู้ใช้ยืนยันและเลือก Claim (Selective Disclosure)
แอปแสดงว่าจะเปิดเผยข้อมูลอะไรบ้าง ผู้ใช้กดยืนยัน — SD-JWT VC ให้เลือกได้ว่าจะส่ง claim ไหน เช่น แสดงแค่ "อายุ ≥ 20" โดยไม่เปิดวันเกิดจริง
Library: @sd-jwt/core
User Consent
4
สร้างและเซ็น VP Token
สร้าง Verifiable Presentation รวม credential ที่เลือก แล้วเซ็นด้วย hardware key — ต้องผ่าน biometric
Hardware Key
5
ส่ง Presentation Submission กลับ Verifier
ส่ง vp_token + presentation_submission ผ่าน response_mode=direct_post ไปยัง Verifier endpoint
Network → Verifier
6
NFC Proximity — ISO 18013-5 mdoc
กรณีแตะมือถือกับเครื่องอ่าน NFC — ใช้ proximity mdoc exchange ตาม ISO 18013-5 ถ่ายโอนเฉพาะ claim ที่ขอ ไม่ต้องใช้อินเทอร์เน็ต iOS ใช้ BLE engagement เป็น fallback
NFC / Hardware

กฎความปลอดภัยที่ไม่มีข้อยกเว้น

🔐
Private key ไม่เคยอยู่ใน JavaScript

Key อยู่แค่ใน hardware chip — JS เรียกผ่าน alias ชื่อ etda_wallet_signing_key เท่านั้น byte จริงไม่เคยถึง runtime

👆
Biometric ทุกครั้งที่เซ็น

บังคับระดับ hardware ทั้งตอนรับและแสดงเอกสาร — ไม่มี software bypass ยกเลิก biometric = operation ยกเลิกทันที

🚫
ห้ามใช้ AsyncStorage

บล็อกระดับ lint + CI — ข้อมูลทั้งหมดต้องผ่าน MMKV ที่เข้ารหัสเท่านั้น

🏭
Backend ไม่เคย proxy OID4VCI / OID4VP

Company server แค่ยืนยันตัวตนผู้ใช้และส่ง offer URL กลับมา — issuance และ presentation ทำตรงระหว่างอุปกรณ์กับ Issuer/Verifier เสมอ

📋
Holder DID เป็นของตัวเอง — Self-Sovereign

did:key derive จาก P-256 public key ของอุปกรณ์ — ไม่ต้องพึ่ง server ใดในการ resolve หรือยืนยัน

👁
Selective Disclosure — เปิดเผยเฉพาะที่จำเป็น

SD-JWT VC ให้ผู้ใช้เลือกได้ว่าจะเปิด claim ไหน เช่น แสดงแค่ "อายุ ≥ 20" โดยไม่เปิดวันเกิดจริง — ผ่าน @sd-jwt/core

🕵
ตรวจสอบ Issuer ผ่าน did:web ก่อนเสมอ

ทุก VC ที่รับมาต้องผ่านการ resolve did:web และตรวจลายเซ็น Issuer บนอุปกรณ์ก่อนเก็บ — ป้องกัน credential ปลอม
